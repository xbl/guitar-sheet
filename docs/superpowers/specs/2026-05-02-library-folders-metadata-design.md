# 谱库：文件夹、元数据与拖拽整理

**状态**：已定稿（待读者过目后可与主规格交叉修订目录表述）  
**日期**：2026-05-02  
**依赖**：[Guitar Sheet — 产品设计说明](./2026-05-02-guitar-sheet-design.md)（MVP 主规格）  

---

## 1. 背景与目标

在 MVP「一条索引记录对应一份谱子文件」的基础上，增加：

- 用户可创建**任意深度**的文件夹，对谱子归类；**本地磁盘目录结构与文件夹树一致**（见第 3 节）。  
- 每条谱子具备可检索的元数据：**曲谱名称（必填）**、歌手（可选）、自由标签（可选多个）；支持按名称 / 歌手 / 标签检索。  
- **应用内**拖拽管理文件夹与谱子；支持从**访达**拖入支持的文件类型，导入到**目标文件夹**对应磁盘路径。  
- 若启用 GitHub 同步：**远端仓库中的 blob 路径与本地相对路径一致**（在用户配置的子目录前缀之下），与主规格「一条记录对应一个远端 blob」一致。  

**非目标（本期）**：将谱子拖出到访达导出；合唱 / 多歌手字段；协作编辑（仍以主规格冲突策略为准）。

## 2. 与主规格的关系

- **冲突检测、三选一策略、Token 安全、同步按钮语义**：完全遵循主规格第 6、7 节，本文只补充**路径如何随文件夹变化**。  
- 主规格第 5.1 节「内容区可采用 UUID 目录或扁平命名」在启用本规格后调整为：**内容区按文件夹树形成真实子目录**；索引中 `local_path` 仍为相对应用数据根的路径，且与实际文件位置一致。

## 3. 存储与目录策略（思路 1：磁盘目录 = 文件夹树）

### 3.1 逻辑与物理

- 存在一棵 **文件夹树**（根下可有零层或多层子文件夹）。  
- **物理内容区**在应用数据目录下有一段受控根路径（例如 `library/content/`），其下目录层级与**文件夹树**一一对应：每个逻辑文件夹对应磁盘上**一个目录**；每个谱子文件存放在**其所属文件夹**对应的目录内。  
- **移动谱子**：在目标文件夹目录内执行文件 `rename`/`move`，并更新索引中的 `local_path`、展示用元数据及由路径推导的 `remote_path`（见 5.2）。  
- **移动文件夹**：对整个子树做目录级移动（`rename` 父目录或递归移动，由实现选定），并**批量更新**该子树下所有谱子（及子文件夹）在索引中的路径相关字段；任一环节失败应回滚或恢复一致状态（实现计划细化事务与错误提示）。  
- **新建空文件夹**：创建对应磁盘目录；索引中插入文件夹记录。若实现侧不允许零字节目录在某些 FS 上残留问题，可在实现计划中约定占位策略（须**不向 GitHub 推送无谱子意义的垃圾文件**，或以 `.keep` 类占位且同步层明确排除——实现计划定稿）。  

### 3.2 命名与安全

- 文件夹名、磁盘段名需 **sanitize**（禁止路径穿越、控制字符；与主规格「避免路径注入」一致）。  
- 同一父文件夹下**文件夹名称唯一**；同一文件夹内谱子**展示文件名**冲突时，导入或重命名规则需给出明确策略（自动后缀或禁止保存），避免覆盖磁盘文件。

## 4. 数据模型（索引扩展）

在 SQLite（或主规格选定库）中至少增加以下概念（字段名可在实现中映射为 snake_case）：

### 4.1 `folders`

| 字段 | 说明 |
|------|------|
| `id` | 主键 |
| `parent_id` | 父文件夹；**`NULL` 表示谱库根下的顶层文件夹**（磁盘上为内容区根目录下的第一级子目录；内容区根路径由实现固定且禁止用户注入） |
| `name` | 展示名；sanitize 后对应磁盘目录名 |
| `created_at` 等 | 可选 |

### 4.2 `sheets`（在主线字段上扩展）

主规格已有：`id`、`kind`、`local_path`、`local_content_hash`、`remote_path`、`remote_blob_sha`、时间戳等。  

本规格**增补 / 明确**：

| 字段 | 说明 |
|------|------|
| `folder_id` | 所属文件夹 |
| **`display_title`（曲谱名称）** | **必填、非空 trimmed 字符串**；用户可在属性中编辑；默认可自导入文件名推导，但保存时不得为空 |
| `artist` | 可选；单行文本，表示歌手或演奏者（个人自用单值即可） |
| `kind` 等 | 与主规格一致 |

### 4.3 标签

- **`tags`**：`id`、`name`（规范化唯一，例如折叠连续空白、大小写规则固定一种）。  
- **`sheet_tags`**：`sheet_id`、`tag_id` 多对多。

## 5. GitHub 路径与移动后的同步语义

### 5.1 `remote_path` 构成

- `remote_path` = 用户配置的**同步子目录前缀** + 对应谱子在**逻辑树中的相对路径**（含各级文件夹名 sanitize 结果）+ `/` + **谱子文件在磁盘上的存储文件名**（与 `local_path` 末端一致）。  
- 移动谱子或文件夹后，本地 `local_path` 更新完成即视为「下次同步应以此路径为准」；与远端对齐时遵循主规格冲突判定（按路径与 blob）。  

### 5.2 路径变更与远端

- 用户在应用内完成移动且磁盘与索引一致后，同步行为等价于：远端「旧路径 blob」与「新路径 blob」的差异处理——实现可采用删除远端旧路径 + 上传新路径等 REST 操作组合，**不得静默覆盖**用户未确认的冲突情形（主规格策略 B）。

## 6. UI / 交互

### 6.1 双入口（已确认方案 C）

- **侧栏**：嵌套文件夹树；选中节点时在主区域列出该文件夹内谱子（含子项与否：默认**仅当前文件夹**，不在此规格强制「包含子树」列表，避免与树导航重复；若实现提供「包含子文件夹」开关，放在实现计划）。  
- **「全部谱子」**：扁平列出库内全部谱子，配合搜索 / 筛选（歌手、标签、名称）。入口形式（侧栏「全部」节点或 Tab）由实现计划选定一种默认。  

### 6.2 拖拽

- **应用内**：在侧栏或列表中拖拽谱子条目、文件夹节点，将条目移动到另一文件夹或根；目标高亮；失败时提示原因。  
- **拖到「全部」**：视为将谱子或文件夹移动到**根级**（与根下并列），具体根语义与 `folders` 根表示一致。  
- **访达拖入**：将支持的扩展名拖至窗口或某一文件夹；复制文件到该文件夹对应目录，创建索引记录；**曲谱名称**默认取自文件名，用户可在导入确认步骤或属性中修改，**保存库时必须非空**。  

### 6.3 检索

- 搜索范围至少包含：**曲谱名称**、**歌手**、**标签**（子串匹配即可；大小写规则与标签规范化一致）。  
- 「全部谱子」视图与带搜索的谱库首页共享同一套过滤语义（实现可合并为一个组件）。  

---

## 7. 错误处理（补充）

| 场景 | 期望行为 |
|------|----------|
| 移动时磁盘失败（空间不足、权限） | 不提交索引变更；提示用户；库状态与移动前一致 |
| 导入后曲谱名称为空 | 阻止完成入库；焦点置于名称输入 |
| 重命名文件夹导致路径过长 | 拒绝或截断策略在实现计划中写明并提示 |

---

## 8. 测试要点（本规格）

- 深层嵌套下创建、移动、删除文件夹；跨层拖拽谱子；冲突文件名规则。  
- 访达拖入到指定文件夹后，`local_path` 与树一致；**曲谱名称**必填校验。  
- （若已接同步）移动后 `remote_path` 更新与冲突流程仍符合主规格。
