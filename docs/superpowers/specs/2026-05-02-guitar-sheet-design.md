# Guitar Sheet — 产品设计说明（初学者吉他谱桌面应用）

**状态**：已评审（与产品负责人对齐）  
**日期**：2026-05-02  
**范围**：第一期（MVP）  
**技术栈**：Tauri 2、Vue 3、TypeScript；同步层采用 **GitHub REST API + 本地索引**。

---

## 1. 背景与目标

### 1.1 产品定位

面向**初学者**的**吉他谱桌面应用**，以**看谱、练谱**为第一优先级：大字号、清晰排版、稳定离线体验；可选将谱子同步到用户**自建的 GitHub 仓库**。

### 1.2 成功标准（MVP）

- 用户可**导入**谱子并在应用内**稳定打开、阅读**；亦可按 [谱库：文件夹、元数据与拖拽整理](./2026-05-02-library-folders-metadata-design.md) **新建文本 TAB** 并在应用内编辑保存（与导入条目在存储与同步上一致）。  
- 未配置 GitHub 时，谱子**完全本地**可用。  
- 配置 Token 与仓库后，可进行**双向同步**；出现潜在冲突时**不静默覆盖**，而是**提示用户**在「保留本地 / 使用远端 / 另存为新文件」中三选一。  
- Token **不以明文**写入谱子文件或可被误提交的配置中；优先使用系统级安全存储（macOS 钥匙串等）。

---

## 2. 非目标（第一期不做）

- Guitar Pro、MusicXML 等**专业谱面格式**的解析与渲染。  
- **复杂练习模式**（段落循环、跟伴奏等）、音频回放（可作为后续迭代）。  
- **节拍器与谱面自动滚动**：见专项规格 [练习模式：节拍器 + 谱面自动滚动](./2026-05-02-practice-auto-scroll-design.md)（简单档位滚动 + BPM 节拍器，可与阅读视图分期上线）。  
- 多人实时协作编辑（同一文件多人同时改）——不优化该场景，仅通过冲突策略**安全降级**。  
- 底层采用 **libgit2 / 捆绑 git 二进制** 的完整 Git 客户端（保留为远期选项；第一期选定 **REST + 索引**）。

---

## 3. 用户与典型场景

### 3.1 主要用户

- 刚入门、以**跟谱练习**为主的吉他初学者。  
- 希望谱子**掌握在自己手里**（本地 + 自己的 GitHub），而非绑定单一商业谱库。

### 3.2 典型流程

1. **添加谱子**：**导入**（从文件系统选择文本 TAB、PDF 或图片），或 **新建文本 TAB**（应用内创建空白/模板文件并编辑正文），进入本地谱库；文件夹归类、元数据、拖拽与自建流程见 [谱库：文件夹、元数据与拖拽整理](./2026-05-02-library-folders-metadata-design.md)。  
2. **阅读**：在「阅读视图」中调整字号/行距；对 PDF/图片进行缩放与翻页或连续滚动。  
3. **（可选）同步**：在设置中配置 GitHub Token、仓库、分支、可选子目录；执行「同步」从远端拉取更新并推送本地变更；遇冲突按第 6 节处理。

## 4. MVP 功能规格

### 4.1 导入与类型

| 类型 | 扩展名 / 说明 | 第一期行为 |
|------|----------------|------------|
| 文本 TAB | `.txt`、`.md`、可无后缀；默认 **UTF-8**，可提供编码选择（若实现成本过高，可先 UTF-8 + 失败时提示） | 等宽渲染、字号/行距可调 |
| PDF | `.pdf` | 以**看图/看页**方式渲染：缩放、翻页或连续滚动（具体交互在实现阶段定一版默认） |
| 图片 | `png` / `jpg` / `jpeg` / `webp` | 同上，缩放与浏览 |

导入时：将文件**复制**到应用数据目录下的谱库内容区，并在索引中登记元数据（见第 5 节）。**新建文本 TAB** 则在选定文件夹下直接创建 UTF-8 文件并登记索引，详见 [谱库：文件夹、元数据与拖拽整理](./2026-05-02-library-folders-metadata-design.md) §6.4。

### 4.2 谱库（本地）

- **组织与自建**：嵌套文件夹、标签、歌手、检索、拖拽整理及 **新建文本谱** 见 [谱库：文件夹、元数据与拖拽整理](./2026-05-02-library-folders-metadata-design.md)。  
- **列表**：展示已导入条目；支持按**文件名 / 标题**搜索（标题可由用户编辑或默认取自文件名）。  
- **打开**：进入阅读视图；记录「上次打开」「上次滚动位置」为可选增强（若排期紧张可第二期再做）。  
- **删除**：从索引移除并删除本地副本（需确认对话框）。

### 4.3 阅读视图

- **文本**：等宽字体、字号滑块、行距；支持横向/纵向滚动；文本谱（含 **自建**）须支持应用内**编辑与保存**，与 [谱库规格](./2026-05-02-library-folders-metadata-design.md) §6.4 一致。  
- **PDF / 图片**：缩放控件；PDF 支持页级导航。  
- **统一**：顶部或侧栏提供返回谱库入口。  
- **壳层与主题**：应用外壳、默认窗口形态、置顶与主题扩展见 [壳层与主题 UI](./2026-05-02-shell-theme-ui-design.md)。  
- **练习辅助（可选分期）**：节拍器与谱面自动滚动见 [练习模式：节拍器 + 谱面自动滚动](./2026-05-02-practice-auto-scroll-design.md)，以不阻塞 MVP 核心路径为前提可并行或后续排期。

### 4.4 GitHub 同步（可选）

#### 4.4.1 设置项

- **Personal Access Token（PAT）**：写入安全存储；不在仓库与日志中明文打印。  
- **仓库**：`owner/repo` 形式（或等价解析自 URL）；校验格式错误时即时提示。  
- **分支**：默认 `main`，允许用户修改。  
- **子目录**：可选，例如 `sheets/`；所有同步文件均相对于该前缀。

#### 4.4.2 权限与文档

- 在 README 或应用内帮助中说明：私有仓库通常需要含 `repo` 的权限范围；鼓励**最小权限**原则。  
- Token 失效、403、404 时给出**可理解**的错误与修复建议（重新生成 Token、检查仓库名与权限）。

#### 4.4.3 同步语义（第一期）

- 提供至少一个明确的用户动作：**「与 GitHub 同步」**（内部实现可为「先拉后推」或「对账后双向」，但对用户只暴露一个主按钮 + 可选「仅拉取」「仅推送」是否出现由实现计划决定；**不得**在无确认下覆盖用户数据）。  
- **首次同步**：若远端子目录不存在，可按策略创建占位路径或在首次推送时创建文件树（实现需处理空仓库）。  
- **离线**：无网络时仍可阅读本地已导入内容；同步按钮提示网络不可用。

## 5. 数据模型与本地存储

### 5.1 目录布局（逻辑）

- **内容区**：保存导入文件的副本（保持原扩展名；内部 ID 可使用 UUID 目录或扁平命名，由实现选定，需避免路径注入）。  
- **索引**：SQLite（推荐）或等价嵌入式库，字段至少包括：

  - `id`（内部主键）  
  - `display_title`  
  - `kind`：`text` | `pdf` | `image`  
  - `local_path`（相对应用数据根的路径）  
  - `local_content_hash`（用于检测本地是否相对上次同步有修改；算法 SHA-256）  
  - `remote_path`（在 GitHub 仓库中的路径，含子目录前缀）  
  - `remote_blob_sha`（最近一次成功对齐远端后的 blob SHA；无同步过则为空）  
  - `last_local_modified_at` / `last_synced_at`（RFC3339 或 Unix 毫秒）

### 5.2 与 GitHub 的映射

- 一条索引记录对应远端**一个 blob 文件**（一对一）。  
- 重命名：视为「删除旧远端路径 + 上传新路径」或仅本地改名不同步，直到用户再次同步；具体策略在实现计划中写清，避免静默分叉。

## 6. 冲突检测与用户决策（已锁定：策略 B）

### 6.1 检测思路（文件粒度）

在「拉取」或「合并对账」时，对某一 `remote_path`：

1. 若本地**自上次记录的 `remote_blob_sha` 以来**内容 hash 已变，且远端 blob SHA 与本地记录的 `remote_blob_sha` **不一致**，则判定为**潜在冲突**（本地与远端均已相对共同基线发生演变）。  
2. 若仅远端变、本地未变：可直接更新本地文件与索引。  
3. 若仅本地变、远端未变：可直接推送。  
4. 若判定为潜在冲突：**停止自动写入**，弹窗提供：  
   - **保留本地**：以本地内容推送（覆盖远端该路径；需二次确认文案）。  
   - **使用远端**：用远端内容覆盖本地副本并更新索引与 hash。  
   - **另存为新文件**：本地复制为新 `remote_path`（或新显示标题），避免覆盖；索引新增一条。

### 6.2 边界

- 不保证「多人同时编辑」体验；仅保证**不丢数据、不静默覆盖**。  
- 极大文件（例如超大扫描 PDF）的同步与超时策略：第一期以**清晰错误提示 + 可重试**为准，可不优化分块上传（GitHub 单文件限制需在帮助文档中提示）。

## 7. 架构概览

### 7.1 分层

- **前端（Vue）**：谱库 UI、阅读视图、设置页、同步与冲突对话框；通过 Tauri `invoke` 调用命令。  
- **后端（Rust）**：文件 IO、SQLite、安全存储读写、GitHub REST 调用（HTTP 客户端）、同步状态机。  
- **网络**：仅在与用户显式触发同步或测试连接时访问 `api.github.com`。

### 7.2 GitHub API（第一期）

- 使用 **REST v3**（或官方推荐的最新稳定 REST 路径）完成：  
  - 获取目录内容 / 获取 blob / 创建或更新文件内容（携带 `sha` 做更新）。  
- 所有请求携带 `Authorization: Bearer <token>` 与合适 `Accept` 头。  
- 注意 **rate limit**：在 UI 或日志中提示剩余额度不足时的降级行为。

### 7.3 安全

- Token：**安全存储**（macOS Keychain；实现可选用 Tauri 生态插件或 `keyring` crate）。  
- **日志**：禁止打印完整 Token；错误信息脱敏。  
- **CSP / 前端**：保持 Tauri 默认安全基线；对外仅 GitHub API。

## 8. 错误处理（第一期必须具备）

| 场景 | 期望行为 |
|------|----------|
| Token 无效 / 过期 | 明确错误 + 引导重新生成并粘贴 |
| 仓库无权限 / 404 | 检查 `owner/repo`、Token 权限、仓库是否存在 |
| 网络超时 | 可重试；本地阅读不受影响 |
| 导入损坏 PDF | 提示无法打开；不崩溃 |
| 远端文件被手动删除 | 同步时按「远端不存在」分支：可提示用户选择重新上传或删除本地映射（实现计划细化） |

## 9. 测试策略（MVP）

- **单元测试（Rust）**：路径安全、hash 计算、冲突判定纯函数。  
- **集成测试（可选）**：对 GitHub API 使用 mock 服务器或录制 fixture（避免 CI 依赖真实 Token）。  
- **手动测试清单**：导入三类样例、离线阅读、同步无冲突、冲突三操作、Token 错误、大文件失败提示。
