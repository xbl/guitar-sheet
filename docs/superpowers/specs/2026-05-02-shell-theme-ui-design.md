# 壳层、窗口形态与主题 UI — 设计规格

**状态**：草案（与产品负责人对齐用）  
**日期**：2026-05-02  
**依赖**：[Guitar Sheet MVP 总规](./2026-05-02-guitar-sheet-design.md) 中「阅读视图」「谱库」等能力描述；本规格约束**桌面壳层、默认窗口、置顶、视觉主题与响应式布局**，不改变第一期同步与导入的业务语义。

---

## 1. 背景与目标

### 1.1 问题

当前界面偏「网页文档」：全局导航与各页顶栏叠用、谱库居中窄版、窗口默认较小，**谱面可视区域**与**桌面应用一体感**不足。

### 1.2 目标

- **应用气质**：默认采用 **明亮纸张（Light paper）** — 浅灰壳层、白/暖白阅读面、适合日间长时间看谱。  
- **屏幕利用**：启动即尽可能用大窗口；**阅读区**占满窗口在壳层之下的剩余空间。  
- **像原生 app**：**单一应用顶栏（chrome）**，避免多段重复横条挤压内容区。  
- **练琴场景**：提供 **置顶（图钉）** 开关，便于谱窗叠在宿主或其它窗口之上。  
- **演进**：主题与颜色以 **CSS 变量 + `data-theme`** 为扩展点，为后续「用户自定义 / 多套预设」留接口，本期不实现完整主题 marketplace。

### 1.3 成功标准

- 用户首次打开应用时，主窗口为 **最大化**（仍保留系统标题栏与常规窗口管理）。  
- 阅读页中，曲谱（文本 / PDF / 图片）所在容器在常见分辨率下 **明显大于改版前** 默认可用区（通过合并 chrome + 弹性布局达成，具体像素随窗口变化）。  
- 窄屏宽度下，顶栏控件 **折行或收入溢出菜单**，不出现横向不可达控件。  
- 图钉可在「置顶 / 非置顶」间切换，状态 **持久化** 至下次启动。  
- 视觉样式由 **设计 token（CSS 变量）** 驱动，便于增加第二套内置主题或后续用户覆盖。

---

## 2. 非目标（本期不写进实现验收）

- **启动即强制系统全屏**（独占全屏）：不作为默认要求；若实现「一键全屏」为增强项，与默认最大化并存即可。  
- **主题插件市场、任意第三方 JS 主题**。  
- **完全自定义标题栏（无边框自绘标题栏）**：本期不强制；若平台默认标题栏保留，图钉放在应用自绘 chrome 内即可。

---

## 3. 窗口与启动形态

| 项 | 规格 |
|----|------|
| **默认** | 主窗口 **启动时最大化**（`tauri.conf.json` 或启动后等价 API，以实现时 Tauri 2 文档为准）。 |
| **可选全屏** | 可提供 **一键进入/退出系统全屏** 的菜单或快捷键；**不**作为默认启动形态（与产品负责人确认：非必须）。 |
| **阅读优先** | 最大化与（若存在）全屏切换均不得破坏谱库索引数据；仅影响窗口 chrome。 |

---

## 4. 壳层与信息架构

### 4.1 单一 chrome

- **合并** 当前「全局 `nav` + 各页独立顶栏」为 **一层** 应用 chrome：品牌或当前上下文标题、导航（谱库 / 当前页）、**设置**、**置顶图钉** 等。  
- **子页面**（谱库工具条、阅读器返回与字号等）作为 **同一顶栏内的区块** 或 **顶栏第二行（仍计为一层 chrome，高度可控）**，避免再出现与 `App.vue` 重复的第三条全宽导航。  
- **阅读页**：返回谱库、文档标题（截断）、图钉、设置入口；**字号 / 行距 / PDF 翻页** 等与阅读相关的控件与上述同区布局，**窄屏**时次要控件进入 **「⋯」溢出菜单**。

### 4.2 主内容槽（阅读 / 谱库列表）

- 应用根布局使用 **flex 列**：`chrome` 固定高度，`main` 为 `flex: 1; min-height: 0; overflow: auto`（或按路由子树传递），保证 **仅内容区滚动**，避免双重 `100vh` 导致滚动条异常。  
- **阅读器**：谱面区域在 `main` 内占满剩余空间；文本 `pre`、图片、PDF 画布均在可滚动容器内自适应 **max-width: 100%** 与合理 **padding**（token 控制）。

### 4.3 谱库布局与响应式

- **宽屏**：列表可采用 **多列栅格** 或宽列表，避免内容挤在固定 `48rem` 内。  
- **窄屏**：单列；工具按钮 **换行**；标题与操作不溢出视口。  
- **断点**：不在本规格写死像素表；实现阶段用 1～2 个断点（如「窄 / 宽」）并在实现计划中列出具体值。

## 5. 明亮纸张主题（默认）

### 5.1 Token 策略

- 在根节点（如 `html` 或 `#app`）使用 **`data-theme="light-paper"`**（名称可微调，实现一致即可）。  
- 颜色、边框、表面背景、主按钮、链接、错误/成功提示色等一律使用 **`var(--…)`**，禁止在组件内散落硬编码色值（迁移期允许逐步替换）。  
- **阅读面**背景与 **壳层**背景应有可辨层级（例如壳 `#f4f4f5`、阅读面 `#ffffff` 或极浅暖白），保证「纸张」隐喻。

### 5.2 后续内置主题

- 预留 **`data-theme`** 其它取值（如未来暗色）；切换时仅改属性与变量，不重写页面结构。

## 6. 置顶（图钉）

### 6.1 交互

- 在 chrome **显著位置**提供 **图钉图标按钮**：切换「始终在最前」。  
- **开启态**与 **关闭态** 有明确视觉区分（如图标填充 / 背景强调）。  
- **工具提示**（`title` 或等价）：说明「固定在最前 / 取消固定」。

### 6.2 实现要点

- 使用前端 **`@tauri-apps/api/window`**：`getCurrentWindow().setAlwaysOnTop(enabled)`（API 名称以实现时官方文档为准）。  
- 在 Tauri **capabilities** 中为该窗口增加 **官方文档要求的 `setAlwaysOnTop` 相关权限**（实现时核对 `core:window:…` 具体标识并写入 `capabilities`）。  
- **持久化**：将布尔值写入应用偏好存储（与现有设置同一持久化机制；若尚无统一位置，实现计划中选择 SQLite 或应用数据目录下 JSON，二选一并文档化）。  
- **已知边界**：若窗口执行 `hide` 后再 `show`，部分平台可能丢失置顶状态 — 规格要求在 `show` 后 **按持久化偏好重新应用** 置顶。Linux Wayland 等环境可能无效或表现不一致：失败时 **向用户显示简短提示**，不静默失败。

## 7. 主题扩展与自定义（分期）

| 阶段 | 内容 |
|------|------|
| **本期（与壳层改版同交付）** | 完整落地 **light-paper** token；结构支持 **`data-theme`**；偏好中可预留「主题 id」字段（默认 `light-paper`）。 |
| **第二期** | 用户 **`theme-overrides.json`**（仅允许白名单 token 键）或应用数据目录下 **`user-theme.css`**（限定仅覆盖 `:root` / 主题变量，文档说明安全边界）。 |
| **第三期** | 多内置预设、主题导入/导出；仍 **不** 承诺任意执行第三方脚本。 |

---

## 8. 与总规的衔接

- [Guitar Sheet MVP 总规](./2026-05-02-guitar-sheet-design.md) §4.3 **阅读视图** 中「顶部或侧栏提供返回谱库」仍成立；本规格将「顶部」细化为 **单一 chrome 内的返回与标题**，并增加置顶与主题 token。  
- 文本 TAB 编辑、PDF/图片缩放与冲突处理等 **不因本规格改变业务规则**。

---

## 9. 测试与验收提示（实现计划可展开）

- 启动窗口是否为最大化；阅读页主内容区是否随窗口缩放。  
- 调整窗口至窄宽度：顶栏是否无截断死控件（溢出菜单或折行）。  
- 图钉开关：置顶行为、重启后恢复；在模拟 `hide`/`show` 后是否仍符合 §6.2。  
- 切换（若已实现）主题属性后，主要屏幕无未定义变量导致的「透明/继承错乱」。
